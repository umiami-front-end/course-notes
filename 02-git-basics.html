Git Basics

What is version control?

Software (and the process) which keeps track of successive revisions. A user can refer to old versions of work if necessary.

Single File vs. Multiple Files

Single files may be tracked using a naming convention (i.e. v1, v2, etc.).

What if a change involves many, related files? How do you group those changes together? What if a project has thousands of files?

Repository

A repository is a project in which all work is stored and tracked.

A repository's contents include files, structure of directories and hidden files which keep track of all changes.

Local repositories reside on a developer's computer; remote repositories are on a server.

  1. A file's previous version.
  2. A current version which is changed.
  3. A (new) saved version which replaces the previous version.

The "commit"

Git is concerned with the commit (saved state) than the actual changes. Each commit represents a snapshot of the project at a given time.

"Commit or it didn't happen."

CLI vs. GUI

Git's "natural" environment is the CLI, though GUI programs also exist.

We will be working with the command-line version of git.

Git Commands

  • More than 75 functions
  • Require at least one other argument
    • git log
    • git checkout master
  • Can be read like a sentence
    • "Git, show me the log."
    • "Git, checkout the 'master' branch."

Installing Git (on macOS)

Typing git at the command prompt will alert you to install "command line developer tools."

A GUI install package is available at git-scm.com for all operating systems.

Configuring Git

Git tracks commits and who made the commits.

$ git config --global user.name "Jane Doe"
$ git config --global user.email "jane@email.com"
$ git config --global core.editor "nano"

The --global flag sets these configurations for all projects, but a different configuration may be set for just one project.

Initializing a Repository

Git has to know about the files to be track before it can recognize a change. To do this, initialize the repository.

git init

Cloning a Repository

Another way to obtain a repository is to clone one from a server (such as GitHub). This saves a replica of the project on your computer.

git clone <url>

Preparing a Commit

To create a commit, first identify which files should be included in the snapshot. This is called staging.

git add <filename(s)>

Checking Status

How do we know which files are being tracked? Modified? Ready to be committed? We can check the repository's status.

git status

Commit!

A commit locks down the files we have added to the staging area.

git commit --message "A well-crafted commit message!"

When to Commit?

A user's workflow may vary, but ideal commits are done when the code for the feature being worked on functions correctly.

For example: If you are creating a web page, a commit may occur when the HTML document structure has been created and validates.

What to commit?

Again, the contents of a commit may vary from change to change. The change may include a change to one line in a single file or multiple lines in multiple files.

For example: If you are adding a style sheet to a web site, you would include the css file, and any html files which had the <link> tag added.

Writing Commit Messages

When writing commit messages, be:

  • Useful. Describe what changed.
  • Detailed (enough). Keep commit messages to ~70 characters.
  • Consistent. Pick a voice, stick with it.

A Repository's History

With every commit, we create a history of our project.

When working with git, we are always moving forward!

View a Repository's Log

To view a repository's history:

git log
git log --pretty=oneline
  • shows every commit (in reverse order)
  • can check differences between branches
  • can check a range of commits using ids
  • a complete list of options (and why to use them) available online

Removing Files

The removal of files is also tracked.

git rm <filename>

After this change is committed, the next (and future) snapshot(s) will not include the file.

If a file is removed via the GUI, git is still tracking the file!

git rm does nothing if the file has already been removed.

Examining Differences

Git can display differences between files, commits and branches.

git diff <filename/branch>
git diff --stat HEAD~1  # one step back in log

Branches

A branch is a copy of a project. This is used for experimentation without fear of damaging the original, master version.

git branch  # lists all branches

Creating a Branch

The branch argument is also used to create a new branch:

git branch <branch-name>

Branch names should be based on its purpose (i.e. fix-typo,
change-font-styles).

Checking out a Branch

Creating a branch does not automatically switch you over to the branch. You must checkout the branch before making changes.

git checkout <branch-name>

When you want to return to the master branch, type:

git checkout master

Other Uses for Checkout

git checkout can also be used to review previous commits. Only do this to review old code, experiment or create new branches.
